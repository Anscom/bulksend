BulkSend · Send Worker Service

Worker Design — Node.js + TypeScript

A KafkaJS worker that turns a campaign job into delivered, tracked email. One deployable, two consumer roles: a dispatcher that fans a campaign out into per-recipient jobs, and a sender that does the rate-limited, idempotent send. Built for at-least-once delivery and crash-safe, distributed completion tracking.

1

Why two roles, not one loop

campaign_jobs → fan-out → email.send

A naive worker would consume campaign_jobs and loop over every contact inline. That can't satisfy your last requirement — “track completion when processing is distributed across workers” — because one message would be pinned to one worker for the whole campaign, and a crash mid-loop would reprocess millions of contacts on redelivery. So the worker splits the job in two:

Steps 1 (fetch contacts) lives in the dispatcher; steps 2–6 (limit, suppress, send, persist, retry) run per-message in the sender so they parallelize across the consumer group; step 7 (mark sent) is driven by a shared atomic counter so whichever worker finishes the last contact closes the campaign. campaign_jobs stays exactly the contract you specified.

2

Role A — Dispatcher

consume campaign_jobs · resumable fan-out
  1. Claim the campaign. Guarded transition scheduled|draft → sending. If no row updates, another worker already claimed it → idempotent no-op.
  2. Stream the audience. Page through the segment query with a keyset cursor — never load millions of contacts into memory.
  3. Create sends rows in bulk with ON CONFLICT (campaign_id, contact_id) DO NOTHING, so a re-run after a crash can't duplicate.
  4. Fan out one EmailSendMessage per created row to email.send, and persist the cursor as a resume point.
  5. Seed the completion counter once with the final recipient total (SET … NX) and store recipient_count on the campaign.
consumer: campaign_jobs · dispatchCampaign()
async function dispatchCampaign({ campaignId, workspaceId }: CampaignJob) {
  // (1) claim — UPDATE ... WHERE status IN ('scheduled','draft') RETURNING *
  const campaign = await db.campaign.markSending(campaignId, workspaceId);
  if (!campaign) return;                  // already dispatched → no-op

  let cursor = campaign.resumeCursor;       // resume after a crash
  let total  = 0;

  // (2) page the segment with a keyset cursor
  for await (const page of streamContacts(workspaceId, campaign.segmentId, cursor)) {
    // (3) bulk insert sends — duplicates are swallowed
    const created = await db.sends.insertMany(
      page.map(c => ({ campaignId, workspaceId, contactId: c.id, status: 'pending' })),
      { onConflict: 'DO NOTHING' },
    );
    total += created.length;

    // (4) fan out + checkpoint the cursor
    await producer.sendBatch('email.send', created.map(toSendMessage));
    cursor = page.at(-1)!.id;
    await db.campaign.saveCursor(campaignId, cursor);
  }

  // (5) seed the completion counter ONCE, now that N is known
  await redis.set(`campaign:${campaignId}:remaining`, total, 'NX');
  await db.campaign.setRecipientCount(campaignId, total);
}                                            // commit offset only after this resolves
WHY

The dispatcher commits its Kafka offset only after the function returns. If it dies mid fan-out, the message is redelivered, markSending is a no-op, and the loop resumes from the saved cursor — already-created sends and already-produced jobs are absorbed by the unique constraint and idempotent sends.

3

Role B — Sender

consume email.send · steps 2–6, per message
  1. Rate limit — atomic Redis token-bucket per workspace. No token? Re-queue with a short delay; this is throttling, not a failure (attempt count untouched).
  2. Suppression check — O(1) SISMEMBER against the unsubscribe/bounce cache; on cache miss fall back to the DB and warm the cache. Suppressed → finalize as skipped.
  3. Send — call SendGrid with the message's idempotencyKey so a provider-side retry can't double-deliver.
  4. Persist — finalize the sends row with status + provider_message_id in one guarded update.
  5. Settle the counter — only a real pending → terminal transition decrements the campaign counter (step 7's trigger).
consumer: email.send · handleSend()
async function handleSend(job: EmailSendMessage) {
  // (2) per-workspace rate limit — atomic, see tokenBucket.lua
  if (!(await rateLimiter.tryAcquire(job.workspaceId))) {
    return requeue(job, '5s', { countsAsAttempt: false });   // throttled, not failed
  }
  // (3) suppression cache (unsubscribed / hard-bounced)
  if (await isSuppressed(job.workspaceId, job.contactId)) {
    return finalize(job, 'skipped');
  }
  try {
    // (4) send via SendGrid (idempotencyKey dedupes provider-side)
    const res = await sendgrid.send(render(job), { idempotencyKey: job.idempotencyKey });
    // (5) persist result + maybe complete the campaign
    await finalize(job, 'sent', res.messageId);
  } catch (err) {
    await onSendError(job, err);            // (6) retry or dead-letter
  }
}                                          // commit offset after handler resolves (at-least-once)
exactly-once settle · finalize()
async function finalize(job: EmailSendMessage, status: SendStatus, messageId?: string) {
  // only the row STILL 'pending' is updated → returns 0 on redelivery
  const changed = await db.sends.finalize(job.sendId, status, messageId);
  //   UPDATE sends SET status=$2, provider_message_id=$3, sent_at=now()
  //   WHERE id=$1 AND status='pending'  →  rowCount
  if (changed === 0) return;               // already settled — stop, do NOT decrement again

  // (7) the decrement is now bound 1:1 to the state transition
  const remaining = await redis.decr(`campaign:${job.campaignId}:remaining`);
  if (remaining <= 0) await completeCampaign(job.campaignId);
}

Error-handling strategy

classify → retry tier → dead-letter

Classify first. The retry budget is precious — never burn it on errors that can't succeed. Transient failures back off and retry; permanent failures settle immediately as a terminal outcome; throttling re-queues without consuming an attempt.

retry router · onSendError()
const MAX_RETRIES = 3;
const BACKOFF = ['5s', '1m', '15m'];

async function onSendError(job: EmailSendMessage, err: unknown) {
  const cls = classify(err);
  if (cls.kind === 'permanent') return finalize(job, cls.isBounce ? 'bounced' : 'failed');

  if (job.attempt >= MAX_RETRIES) {        // exhausted → dead-letter
    await deadLetter(job, err);
    return finalize(job, 'failed');
  }
  const delay = cls.retryAfter ?? BACKOFF[job.attempt];   // transient → back off
  await requeue({ ...job, attempt: job.attempt + 1 }, delay, { lastError: err });
}
At-least-once + idempotent

Manual offset commit after the handler resolves. Redelivery is safe via three guards: UNIQUE(campaign_id, contact_id), SendGrid idempotencyKey, and the WHERE status='pending' finalize.

Delay ≠ sleep

Retries republish to delay topics; the consumer never sleep()s on a partition (that would stall every other message behind it).

Poison messages

Anything that throws before classification (malformed job, repeated deserialization failure) is dead-lettered after its attempts so one bad message can't block the partition forever.

Slow ≠ dead

Bounded batch sizes + tuned max.poll.interval.ms keep a busy worker from being falsely evicted mid-send during a SendGrid slowdown.

5

Distributed completion tracking

step 7 — who marks it “sent”?

With work spread across the consumer group, no single worker knows when the campaign is done. The mechanism is a shared atomic counter plus a guarded transition, backed by a reconciliation sweeper for safety:

  1. Seed. The dispatcher sets campaign:{id}:remaining = N exactly once, after counting the audience.
  2. Decrement on terminal transition only. Every pending → sent|failed|bounced|skipped settle does one atomic DECR. Because the decrement is gated on the guarded DB update (returns 0 on redelivery), it fires exactly once per contact even under at-least-once redelivery.
  3. The worker that hits 0 wins. DECR returns the new value atomically — exactly one worker sees 0 and runs completeCampaign(), itself a guarded sending → sent update so it's safe to call more than once.
  4. Sweeper reconciles drift. A periodic job re-derives truth from Postgres for any campaign stuck sending, covering a Redis eviction or a lost decrement.
guarded close + safety-net sweeper
async function completeCampaign(campaignId: string) {
  // idempotent: only flips if still 'sending'
  const ok = await db.campaign.markSent(campaignId);   // UPDATE ... WHERE status='sending'
  if (ok) await redis.del(`campaign:${campaignId}:remaining`);
}

// runs every ~60s — fixes counter drift (eviction, lost decrement)
async function reconcile() {
  for (const c of await db.campaign.stuckSending({ olderThan: '2m' })) {
    const done = await db.sends.countTerminal(c.id);  // sent+failed+bounced+skipped
    if (done >= c.recipientCount) await completeCampaign(c.id);
    else await redis.set(`campaign:${c.id}:remaining`, c.recipientCount - done); // re-sync
  }
}
KEY IDEA

Redis gives the fast, lock-free common path (one DECR per send, no contention, no COUNT(*) on the hot path); Postgres is the source of truth the sweeper falls back to. The DB transition guard makes the counter exactly-once; the sweeper makes the whole thing self-healing — a campaign can never get permanently stuck in sending just because a counter drifted.

EDGE

A campaign with contacts still cycling through retry tiers stays sending until those settle (success or exhaustion → DLQ + failed). That's correct: “done” means every contact reached a terminal state, not merely “the queue drained.”
