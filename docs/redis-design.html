BulkSend · Cache & Coordination Layer

Redis Key Design

Naming, types, and TTLs for the four hot-path Redis workloads. Every key is tenant-prefixed, every counter mutates through an atomic command (never read-modify-write in app code), and Postgres stays the source of truth — Redis is the fast lane that can always be rebuilt.

Suppression, sessions, and rate-limit policy all originate in Postgres; the Redis keys are caches/coordination state. Anything here can be lost and rebuilt — designs below say how each one recovers so a cold cache or eviction never causes a wrong send or a stuck campaign.

1

Rate limiting — hourly quota

fixed window · atomic reserve

Key convention

rl:{workspaceId}:{YYYYMMDDHH}
e.g. rl:ws_8fa…:2026060415

Type

string int counter, INCR

TTL

7200s, set on creation. Window key self-expires — no cleanup job.

One counter per workspace per clock-hour. The limit (100 / 2000) is resolved live from a tiny cached policy key so a plan upgrade takes effect immediately. The quota is reserved before the send and refunded if the send is skipped, so concurrent workers can't oversend.

reserve → send → refund-on-skip
# resolve the limit (cached from Postgres, short TTL, busted on plan change)
GET ws:{ws}:limit            # → 100 | 2000

# atomically reserve one unit of THIS hour's quota
INCR   rl:{ws}:2026060415     # → n  (atomic; safe across all workers)
EXPIRE rl:{ws}:2026060415 7200 NX   # set TTL only on first write

# n > limit → over quota: requeue the job to the next hour window
# n ≤ limit → proceed to send

# if the send is skipped (suppressed) or permanently fails pre-provider,
# hand the unit back so it isn't wasted:
DECR   rl:{ws}:2026060415
Edge cases
  • Boundary bursts. A fixed window allows up to 2× the limit across the hour seam (full quota at :59 + full quota at :00). If that's unacceptable, switch to a sliding-window log: ZADD rl:{ws} now now → ZREMRANGEBYSCORE … 0 (now-3600) → ZCARD. Costs more memory; gives a true rolling hour.
  • Plan changes mid-window. Never bake the limit into the counter — read ws:{ws}:limit each check and invalidate it on upgrade/downgrade so the new cap is instant.
  • Don't burn quota on non-sends. Reserve before SendGrid, DECR on skip/permanent-fail. Accept a rare tiny overcount if a worker crashes between reserve and send (self-corrects next hour).
  • Clock skew. Derive the window bucket from Redis TIME (or a single API clock), not each worker's local clock, so all workers agree on “this hour.”
  • Two layers. This hourly key is the billing cap; the per-minute tb:{ws} token bucket is separate provider smoothing — a job must pass both.

Unsubscribe cache

O(1) suppression check before every send

Key convention

supp:{workspaceId}
+ supp:{ws}:loaded (flag)

Type

set of contactId

TTL

None — suppression is permanent. Rebuilt from Postgres if evicted.

A per-workspace set of suppressed contactIds gives an O(1) SISMEMBER on the hot path. Compliance is the hard constraint here: a false negative means emailing someone who unsubscribed. So “not in the set” may only be trusted when the set is known-complete — guarded by a :loaded flag.

read (worker) · write (event worker) · rebuild
# --- READ: send worker, before every send ---
EXISTS    supp:{ws}:loaded         # is the cache known-complete?
SISMEMBER supp:{ws} {contactId}     # 1 → skip send (suppressed)
# loaded flag missing → fall back to Postgres AND trigger rebuild (fail-safe)

# --- WRITE: event worker, on unsubscribe / hard bounce ---
SADD supp:{ws} {contactId}          # mirror of contacts.status change

# --- REBUILD: on cold start / eviction ---
SADD supp:{ws} c1 c2 c3 …            # bulk-load all suppressed ids from DB
SET  supp:{ws}:loaded 1             # only now is "not a member" trustworthy

# --- RESUBSCRIBE (rare) ---
SREM supp:{ws} {contactId}
Edge cases
  • Eviction = compliance risk. Put suppression on a noeviction instance/DB (or pin it), so a memory-pressure flush can't silently drop unsubscribes. If it does vanish, the :loaded flag is gone too → the worker fails safe to Postgres until rebuilt.
  • Completeness over freshness. Treat a missing :loaded flag as “unknown,” never “clear.” The flag is the contract that makes a negative answer safe.
  • Huge tenants. Millions of suppressed contacts still fit a set cheaply (tens of MB), but if memory matters use a RedisBloom filter — accept a tiny false-positive rate (occasionally skip a valid send) which is the compliance-safe direction to err.
  • Write ordering. The unsubscribe event must hit Redis before (or atomically with) the DB write, or a send in-flight could slip through. The SISMEMBER + DB fallback covers the race.

Campaign send progress

live counter for the dashboard

Key convention

campaign:{id}:stats (hash)
campaign:{id}:remaining (str)
campaign:{id}:opens · :clicks (HLL)

Type

hash string HLL

TTL

48h, refreshed on write. Final counts persist to Postgres rollups.

A single hash holds the per-status tallies the dashboard polls; HINCRBY is atomic so every worker updates the same fields without contention. Unique opens/clicks use HyperLogLog (PFADD) so we get a cardinality estimate without storing every contactId. remaining is the completion counter from the worker design.

write (workers) · read (dashboard)
# --- WRITE: send/event workers as each result settles ---
HINCRBY campaign:{id}:stats sent 1        # also: failed | bounced | skipped | delivered
EXPIRE  campaign:{id}:stats 172800        # slide the 48h window on activity

PFADD   campaign:{id}:opens {contactId}    # unique opens (HLL ~0.8% error)
PFADD   campaign:{id}:clicks {contactId}

# --- READ: GET /campaigns/:id/stats ---
HGETALL campaign:{id}:stats                # totals by status
PFCOUNT campaign:{id}:opens                # unique open count
# merge with campaigns.* rollups for any campaign older than the 48h window
Edge cases
  • Drift & eviction. The hash is a live mirror, not truth. The reconciliation sweeper (worker design §5) re-derives counts from Postgres for stuck/old campaigns, so a lost HINCRBY never corrupts the final number the user sees.
  • Total vs unique. Keep totals in the hash (one person opening 5× = 5) and uniques in the HLL (= 1). Don't try to derive one from the other.
  • Cold dashboard. Once a campaign finishes and the 48h window lapses, the key is gone by design — the stats endpoint reads the persisted campaigns.sent_count / open_count / … rollups instead.
  • Live updates. Push deltas to the dashboard via SSE/WebSocket rather than hammering HGETALL — or poll on a sane interval (e.g. 2–5s) while status = sending.

Session store

refresh sessions + access-token revocation

Key convention

sess:{sessionId} (hash)
user:{userId}:sessions (set)
auth:bl:{jti} (string)

Type

hash set string

TTL

session 30d sliding; denylist = access-token remaining life (~15m)

Access tokens are stateless JWTs; Redis stores the refresh session (for rotation & revocation) and a short-lived denylist so a logout can kill an access token before it naturally expires. A per-user index set powers “log out of all devices.”

login · refresh · logout · revoke-all
# --- LOGIN: create session + index it ---
HSET    sess:{sid} userId u_… workspaceId ws_… role member rtHash …
EXPIRE  sess:{sid} 2592000           # 30d absolute / sliding cap
SADD    user:{uid}:sessions {sid}     # for device list + revoke-all

# --- REFRESH: validate + rotate (prevents replay) ---
HGET    sess:{sid} rtHash             # compare to presented refresh token
HSET    sess:{sid} rtHash {newHash}   # rotate on every use
EXPIRE  sess:{sid} 2592000           # slide the window

# --- LOGOUT: drop session + denylist the live access token ---
DEL     sess:{sid}
SREM    user:{uid}:sessions {sid}
SET     auth:bl:{jti} 1 EX {tokenTtlRemaining}   # ~15m, then self-expires

# --- REVOKE-ALL (password change / "log out everywhere") ---
SMEMBERS user:{uid}:sessions          # → DEL each sess:{sid}
Edge cases
  • Idle vs absolute expiry. Slide the TTL on refresh for convenience, but enforce a hard maxAge stored in the hash so a session can't live forever just by being used.
  • Orphaned index entries. When sess:{sid} TTL-expires, its id lingers in user:{uid}:sessions (Redis won't auto-remove set members). Lazily SREM on read when the session is gone, or sweep periodically.
  • Denylist only matters until expiry. Checking auth:bl:{jti} on each request gives instant revocation; the key auto-expires when the token would have anyway, so the denylist stays tiny. Skip the check entirely if ~15m of residual validity is acceptable.
  • Token theft / rotation. Rotating rtHash on every refresh means a stolen-then-reused old refresh token fails the compare — detect reuse and revoke the whole session family.
  • Store hashes, not secrets. Persist a hash of the refresh token (rtHash), never the raw token — a Redis dump must not leak usable credentials.
