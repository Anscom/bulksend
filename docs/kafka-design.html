BulkSend · Event Backbone

Kafka Topic Design

Topology, message contracts, consumer groups, and failure handling for the async send pipeline — from “API produces a dispatch” through per-recipient delivery, retries, and event ingestion. Designed for at-least-once delivery with idempotent processing end to end.

1

Topics & why

command / job / retry / dead-letter / event

The API never fans out inline. It writes the sends rows, then produces a single command per campaign. A dispatcher explodes that into one job per recipient, so a 2M-contact campaign doesn't block the request or hold a producer open for minutes. Retries live on separate delay topics (never sleep() on a consumer thread — that stalls the whole partition), and a dead-letter topic catches anything that exhausts them.

KEY

Why key email.send by sendId, not workspaceId? Independent emails need no ordering, and rate limiting is enforced centrally through the Redis token bucket regardless of partition — so a high-cardinality key gives the most even spread and avoids a single large tenant creating a hot partition. Key by workspaceId instead only if you want strict per-tenant ordering or partition-local throttling.

Message payloads

TypeScript contracts

Every message carries an idempotency key and the IDs needed to reconcile against Postgres. Email content is not embedded — workers resolve and cache the rendered template by campaignId (Redis), keeping messages small even across millions of recipients. Only per-contact merge variables ride along.

Consumer group strategy

one group per role · scale ≤ partitions

One consumer group per logical role, scaled horizontally by adding instances up to the topic's partition count. Use cooperative-sticky assignment so adding/removing a worker only moves a few partitions instead of stop-the-world rebalancing the whole group.

SCALE

Adding a 40th send-worker to a 32-partition topic buys nothing — 8 instances sit idle. Scale partitions and instances together, and over-provision partitions at creation (cheap) since you can't easily reduce them later.

Retries & dead-letter queue

max 3 retries · exponential-ish backoff

Classify the error first. Transient failures (HTTP 429, 5xx, timeouts) are retryable; permanent ones (invalid address, hard bounce, suppressed contact) are a terminal business outcome — mark the sends row failed/bounced immediately and don't waste retry budget or DLQ space on them. Retryable failures move the job to the next delay tier rather than blocking the consumer.

  • Delay without blocking. A failed attempt is republished to the next retry.* topic with availableAt = now + backoff. The retry-scheduler reads it; if the delay hasn't elapsed it pause()s the partition (or seeks back) instead of sleeping — so one slow message never stalls others.
  • Bounded retries. attempt is incremented and carried in the payload and persisted on the sends row. After attempt 3 fails, the job is wrapped as a DeadLetterMessage and produced to email.send.dlq; the sends row goes failed.
  • DLQ is for processing failures, not business outcomes. Hard bounces and unsubscribes never enter it. The DLQ holds genuinely stuck jobs (provider outage past the window, malformed data) for inspection and one-click replay back onto email.send.
  • Honor Retry-After. On a 429 from SendGrid, seed availableAt from the provider's header rather than the fixed tier so you back off exactly as long as asked.
5

If a worker crashes mid-campaign

at-least-once + idempotency everywhere

Nothing important lives in worker memory — campaign state is the durable Kafka log plus the sends rows in Postgres. A crash is just a rebalance: the group reassigns the dead worker's partitions to survivors, which resume from the last committed offset. The only hazard is reprocessing the handful of in-flight messages, which idempotency makes harmless.

Commit after the side effect

Use manual offset commits, committing only after the SendGrid call returns and the sends row is updated. Crash before commit → the message is redelivered, never silently lost.

Three idempotency layers

Redelivery is safe because of: the DB UNIQUE (campaign_id, contact_id), the SendGrid idempotencyKey (provider-side dedupe), and providerEventId for events. A reprocessed job is a no-op.

Resumable dispatcher

If the dispatcher dies mid fan-out, it resumes from resumeCursor / its committed offset. Re-emitting jobs it already produced is safe — the unique constraint and ON CONFLICT DO NOTHING absorb duplicates.

Idempotent producer

Run producers with enable.idempotence=true + acks=all so a retried produce after a broker hiccup doesn't write the job twice.

Slow ≠ crashed

Long SendGrid calls can trip max.poll.interval.ms and cause a false eviction. Process in bounded batches and tune the interval, so a busy worker isn't kicked from the group.

Resume, don't restart

Because offsets + sends state are durable, a restarted worker picks up exactly where the group left off. The campaign continues; already-sent recipients are skipped on reprocessing.
